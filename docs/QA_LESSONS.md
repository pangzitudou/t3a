# 经验沉淀（QA / 稳定性）

更新时间：2026-02-17

## 1. 字符集
- MySQL 表结构建议统一 `utf8mb4`。
- JDBC 参数不要写 `characterEncoding=utf8mb4`（部分驱动/连接池不支持），应使用：
  - `useUnicode=true&characterEncoding=UTF-8&connectionCollation=utf8mb4_unicode_ci`
- 脏数据修复前必须先备份受影响行。

## 2. AI 生成链路
- 文件上传后异步处理必须先把 `MultipartFile` 转为字节缓存，否则临时文件可能被容器清理。
- 模型输出不能假设总是严格 JSON，需要：
  - 清洗代码块标记
  - 提取 JSON 数组主体
  - 失败时增加“修复重试”
- 题目数量越大，模型输出越容易截断或格式漂移；应设上限并分批生成。

## 3. 数据模型契约
- 前后端题目模型必须有明确映射层（`questionType/options/correctAnswer/tags`）。
- 列表接口返回给 UI 的统计字段（如 `questionCount`）要由后端显式提供，避免前端猜测。

## 4. 回归策略
- 每次改动后至少跑：
  - API 冒烟（登录/题库列表/题目列表/开考）
  - 页面冒烟（登录→题库→测验→结果）
- 页面检查应包含乱码规则与关键文本存在性检查（题干、选项、结果页）。

## 5. 本轮新增经验（2026-02-17）
- 退出测验“状态更新”不等于“记录删除”：
  - 仅把 `status` 改成 `ABANDONED` 仍会出现在历史列表。
  - 必须同步设置 `deleted=1`（会话 + 答案记录）。
- 题型精确数量配置需要后端强约束：
  - 仅靠 prompt 指令不够，需按题型分批生成，并在去重后自动补题。
- 去重要做两层：
  - 入库前去重（同批/库内题干归一化）
  - 抽题时再去重（避免历史脏数据导致同场测验重复题）
- 本地环境稳定性：
  - JDK 25 可能触发 Lombok 编译异常（`TypeTag :: UNKNOWN`），统一使用 JDK 17。
- 自动化回归建议：
  - 使用 `docs/scripts/e2e-feature-regression.js` 覆盖“登录→题库搜索→答题→结果→退出→生成参数”。
